# План модернизации UI/UX (v1)

Документ фиксирует конкретный план улучшений визуальной части и пользовательского пути.
Цель: повысить читаемость интерфейса, сократить количество действий в ключевых сценариях и подготовить основу для дальнейших итераций.

## 1) Цели и метрики

- Ускорить выполнение ключевых задач (создать/продолжить/завершить тренировку).
- Упростить навигацию в меню для ролей "спортсмен" и "тренер".
- Повысить визуальную иерархию и контраст.
- Убрать дублирующие действия (CTA) и шум в интерфейсе.

Целевые KPI после внедрения:
- Время до старта тренировки: -20%.
- Доля успешного завершения сценария "шаблон -> тренировка -> завершение": +15%.
- Доля использования быстрых действий на главной: >40%.
- Снижение отказов на пустых экранах (group trainings/templates): -20%.

## 2) Приоритеты (что делаем сначала)

### P0 (обязательно, ближайший спринт)

1. Единая визуальная иерархия в боковом меню.
2. Удаление дублирующих CTA на empty state экранах.
3. Блок "Быстрые действия" на главной.
4. Улучшение контраста вторичного текста.
5. Desktop-адаптив для широких экранов (2 колонки после брейкпоинта).

### P1 (следующий спринт)

1. Упрощение структуры меню по сценариям (JTBD).
2. Wizard-поток создания групповой тренировки.
3. Экран итогов после завершения тренировки.
4. Снижение визуального шума в карточках списков (templates/workouts/progress).

### P2 (после стабилизации P0/P1)

1. Микроанимации и состояние интерактивности (hover/pressed/focus).
2. Точечная доработка карточки профиля тренера.
3. Расширение визуальных тестов на UX-проверки (не только скриншоты).

## 3) План работ по направлениям

## 3.1 Визуальная система

### Задача A: Стандартизировать типографику и контраст

Что сделать:
- Зафиксировать роли текста: `title`, `body`, `secondary`, `hint`.
- Поднять контраст `secondary` текста на темном фоне.
- Убрать "серые на сером" сочетания для подписей и вторых строк.

Готовность:
- На основных экранах (`home`, `progress`, `templates`, `drawer`) вторичный текст читаем при первом скане.
- Нет мест, где статус/подпись визуально сливаются с фоном.

### Задача B: Сетка и отступы

Что сделать:
- Принять шаг сетки 4/8pt.
- Базовые правила:
  - внешний отступ контейнера: 16;
  - между блоками: 12;
  - внутренние отступы карточек: 12/16.
- Привести карточки и списки к единому радиусу и плотности.

Готовность:
- Экраны выглядят согласованно без "прыгающих" отступов.

### Задача C: Состояния интерактивных элементов

Что сделать:
- Добавить единые состояния `hover`, `pressed`, `focus`, `disabled` для кнопок и иконок действий.
- Для destructive action (delete) задать отдельный цветовой токен.

Готовность:
- Любой интерактивный элемент имеет предсказуемую реакцию на действие.

## 3.2 Навигация и IA (информационная архитектура)

### Задача D: Перестроить боковое меню

Что сделать:
- Сократить число элементов первого уровня.
- Группировать пункты по задачам пользователя:
  - Спортсмен: `Тренировки`, `Прогресс`, `Связи`, `Календарь`, `Помощь`.
  - Тренер: `Подопечные`, `Групповые`, `Расписание`, `Аналитика`, `Помощь`.
- Вторичные разделы убрать на вложенный уровень.
- У вторых строк оставить только действительно полезные пояснения.

Готовность:
- Пользователь находит нужный раздел за 1-2 шага без чтения всех пунктов.

### Задача E: Улучшить переключение ролей

Что сделать:
- Оставить переключатель ролей в шапке меню, но явно показывать активный контекст.
- При смене роли сохранять last-screen для каждой роли.

Готовность:
- После переключения роли пользователь возвращается в логичную точку, а не теряется в навигации.

## 3.3 Ключевые пользовательские потоки

### Задача F: Быстрые действия на главной

Что сделать:
- Добавить отдельный блок CTA на `home`:
  - `Продолжить тренировку`
  - `Создать тренировку`
  - `Создать шаблон`
- Порядок кнопок адаптировать к состоянию пользователя (если есть активная тренировка, она первая).

Готовность:
- Пользователь запускает ключевой сценарий с главной в 1 клик.

### Задача G: Empty state без дублей CTA

Что сделать:
- На каждом empty state оставлять только один primary CTA.
- Если есть FAB, не дублировать центральную кнопку с тем же действием.
- Добавить короткое объяснение пользы действия (1 строка).

Готовность:
- На пустом экране нет двух одинаковых призывов к действию.

### Задача H: Поток "шаблон -> тренировка -> завершение"

Что сделать:
- После создания шаблона предложить "Сразу начать тренировку".
- На активной тренировке сделать визуально доминирующую кнопку "Завершить".
- После завершения показывать итоговый экран:
  - объем/время;
  - XP/бейджи/PR;
  - CTA: `Сохранить как шаблон`, `Поделиться`.

Готовность:
- Полный цикл выполняется линейно, без возвратов в меню.

### Задача I: Создание групповой тренировки через wizard

Что сделать:
- Разбить сценарий на 3 шага:
  1) Выбор шаблона,
  2) Дата/зал,
  3) Проверка и публикация.
- Добавить stepper и валидации на каждом шаге.

Готовность:
- Пользователь понимает, на каком шаге находится и что осталось заполнить.

## 3.4 Адаптив desktop/tablet

### Задача J: Широкие экраны

Что сделать:
- Ввести брейкпоинты:
  - <900: один столбец;
  - >=900 и <1200: расширенный один столбец;
  - >=1200: две колонки карточек.
- Для `progress` и `home` распределять карточки по колонкам.

Готовность:
- На desktop нет ощущения "узкой мобильной ленты" в центре.

## 4) Технический план внедрения (для разработки)

1. Вынести токены в тему:
   - цвета, контрастные пары, радиусы, отступы, состояния.
2. Добавить/обновить общие UI-компоненты:
   - `AppCard`, `SectionTitle`, `PrimaryCta`, `EmptyStateBlock`, `QuickActionsStrip`.
3. Обновить экраны:
   - `home`, `templates`, `current-workout`, `group-trainings`, `progress`, `main_shell` (drawer).
4. Включить feature-flag для крупных изменений потока (wizard, итоговый экран), если нужно безопасное включение.
5. Доработать интеграционные/визуальные тесты:
   - проверка отсутствия дублей CTA на empty state;
   - проверка наличия "быстрых действий" на home;
   - проверка прохождения полного сценария без ошибок.

## 5) План по спринтам

### Спринт 1 (P0)

- Меню: иерархия, сокращение шума, контраст.
- Empty state: убрать дубли CTA.
- Home: быстрые действия.
- Адаптивные брейкпоинты (базово).
- Обновление визуальных тестов под новые блоки.

Результат:
- Заметно лучше читаемость и скорость старта базовых действий.

### Спринт 2 (P1)

- Wizard групповых тренировок.
- Поток завершения тренировки + экран итогов.
- Дополнительная оптимизация структуры разделов.

Результат:
- Ключевой путь становится линейным и предсказуемым.

### Спринт 3 (P2)

- Полировка micro-interactions.
- Тюнинг профиля тренера.
- UX-метрики и A/B (если применимо).

Результат:
- Финальная консистентность и стабильный UX.

## 6) Критерии приемки

- Нет экранов с дублирующими CTA одинакового действия.
- Основные сценарии доступны с главной за 1 клик.
- Боковое меню сканируется быстро, активный контекст роли очевиден.
- На desktop-контенте нет узкой фиксированной колонки без причины.
- Визуальные тесты проходят, артефакты сохраняются в `tests/visual/...`.

## 7) Риски и ограничения

- Риск: слишком резкое изменение меню может сбить текущих пользователей.
  - Митигировать: постепенный rollout + подсказки при первом открытии.
- Риск: wizard удлинит сценарий для опытных тренеров.
  - Митигировать: добавить "быстрое создание" (advanced mode) после стабилизации.
- Риск: расхождения между mobile и desktop layout.
  - Митигировать: единые токены и компонентный подход.

## 8) Что брать в разработку первым (конкретно)

1. Drawer refactor (иерархия + сокращение пунктов + контраст).
2. Home quick actions.
3. Empty state cleanup в `group-trainings` и смежных экранах.
4. Desktop two-column layout для `home` и `progress`.
5. После этого: wizard групповых и итоговый экран завершенной тренировки.
